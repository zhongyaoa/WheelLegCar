2.3、赛题任务

赛题任务和越野组的赛题任务一致，要求小车在50-100米的跑道长度范围内完成一次往返跑，往返的道路上要求依次通过坡道，圆环的锥桶和S弯道锥桶这些新增元素。车模从出发区驶往调头区，需要经过坡道。在调头区完成掉头之后再重新返回发车区。当遇到圆环锥桶时，需要绕行锥桶一周；遇到 S 弯道锥桶时，需要从锥桶之间间隔一次穿越。


从任务的大框架来看，和极速越野组一样很开放，没有引导线，主要使用GPS或北斗导航，结合IMU做惯导和车身控制，虽然增加了锥桶，但增加的锥桶也正好可以作为特征点来做一些视觉指引。

其实在调试演示车模的过程中也产生了疑问，坡道是否要增加一个视觉提示点来辅助，比如坡道两侧各放置一个锥桶，这也和前面的单个锥桶以及锥桶堆有明显的区别，否则单纯靠卫星定位来让小车准确的找到坡道并完成通过坡道是有难度的，一旦没通过或者一旦跑偏一点，那面临的就是罚时或者撞车，建议规则完善上这一点；还有一个疑问是小车出发后选择先跑锥桶这边还是先跑坡道这边最好能自选，只要能完成往返且按照规则以指定动作通过所有路段即可，这样也增加了赛题的灵活性和宽容度，也能让大家有不同的策略。

3.4、单车越野的导航循迹相关

演示车模的导航主要依靠IMU660RA六轴模块和GPS模块这两个传感器的融合，通过IMU660RA积分和GPS航向角数据可以融合出车模实时的航向角，通过GPS获取当前的定位。根据方向角和自身位置定位、目标位置定位，计算与目标的偏差来控制车模的转向，便能实现导航。接下来我们从位置获取，航向角的融合计算，基本导航算法的实现来展开讲讲。

3.4.1、小车自身航向角

单纯的GPS方位角需要车模移动后才能得到方位角，在车模没有向前行进前，GPS无法提供有效的方位角信息。并且因为GPS的获取解析时间较长，方位角的更新频率无法满足越野车的高速行驶。所以主要使用陀螺仪来计算方位角。

程序中使用定时器来持续获取陀螺仪Z轴的角速度，按照下图，除以相应比例再乘以时间间隔，得到当前间隔内转动的角度。

如设置5ms的定时器中断，在中断函数中读取Z轴角速度值imu660ra_gyro_z，当前间隔转动的实际角速度就是imu660ra_gyro_z / 16.4，然后乘以时间间隔0.005s，得到当前间隔旋转的角度值，将角度值积分，便能得到持续的方位角。


在行驶过程中，车模会产生抖动，如果单纯的使用陀螺仪计算航向角，则一定会导致航向角角出现漂移。因此我们可以使用GPS获取的方向角与陀螺仪的角度值进行互补滤波，来得到较为稳定的航向角，在讲融合之前我们先看看如何处理陀螺仪的零漂，在交流群里看到不少同学在说陀螺仪积分出来的航向角异常的飘，其实正常情况下静止的时候，积分出来的航向角也应该是稳定的。

但实际情况是在陀螺仪静止的时候，它是一个接近于0的值，但并不为0，这个值我们称为零飘。且每一个模块的零飘都会不一样。我们需要对零飘进行处理。实现方法是先让其静止的时候，获取100次陀螺仪数据，每次读取数据后需要延时1毫秒再读取，然后取平均值。我们在每一次获取完陀螺仪数据的时候，减去这个平均值，从而获取到处理后的数据。我们仔细观察这个处理后的数据就能发现，在静止的时候，这个值基本在0附近，如果不在0附近则应该重新采集静态时的零偏数据作为矫正基准。这样处理了可能还不够，因为在静止的时候，每一次获取零飘，他的值都不是固定的，而是在很小的范围内波动，所以，减去了零飘。静止的时候，还是会有一个很小的值。为了去除这样的数据，我们可以将减去零飘后的陀螺仪数据对10取整，去掉个位数，然后再乘以10。这样基本就能保证，在其静止的时候，陀螺仪最后得到的值等于0。



先确保数据准确，然后我们回到前面讲到的可以通过陀螺仪的数据积分得出航向角，但这个航向角只能短时间内稳定，长期的话就存在累计误差。这里就需要GPS模块获得的航向角来辅助了。首先我们知道GPS模块的数据也可以得到航向角，当GPS模块有一定的移动速度后，可以获得稳定的航向角，但更新周期较慢，100ms一次。

所以，我们可以将两者的数据进行融合。这里依然可以利用一阶互补滤波的思想，先将实际的航向角与陀螺仪积分结果相加，得到一个新的航向角，然后与GPS获得的航向角进行对比，如果大于GPS航向角，实际的航向角就增加一点(temp_angle)，如果小于GPS航向角，实际陀螺仪积分出来的角度就减小一点(temp_angle) 。其中的temp_angle需要自己去调试，这个值越大，就更快的趋近GPS的航向角，越小就更慢趋近航向角，具体合适的值需要调试来获得。实现方式可以参考下方的框图：


其中陀螺仪数据/16.4*0.005中的除以16.4是转化为带物理单位的数据，而0.005是5ms的时间，通过这样就可以将其陀螺仪积分数据转化为角度值(0°-360°)。例程的代码注释中也有说明。


航向角稳定、连续，就能让我们的小车跑出较直的直线和连续的曲线，路径好了，速度才上得去，这一个原则在无赛道的极速越野组同样适用。

3.4.3、同步航向角

由于所在的环境不同，GPS的航向角和车模自身的方位角是存在偏差的，所以需要同步GPS和陀螺仪的正方向。这里说两种简单的方法。

第一种，以陀螺仪初始为正方向，方向指向车头。先记录下初始的经纬度定位点，车模向正方向直线行驶，行驶10m - 20m，记录第二个点。可以参考开源库的两点方向角计算方法。

         

//-------------------------------------------------------------------------------------------------------------------

// 函数简介     计算从第一个点到第二个点的方位角

// 参数说明     latitude1       第一个点的纬度

// 参数说明     longitude1      第一个点的经度

// 参数说明     latitude2       第二个点的纬度

// 参数说明     longitude2      第二个点的经度

// 返回参数     double          返回方位角（0至360）

// 使用示例     get_two_points_azimuth(latitude1_1, longitude1, latitude2, longitude2);

// 备注信息

//-------------------------------------------------------------------------------------------------------------------

double get_two_points_azimuth (double latitude1,double longitude1,double latitude2,double longitude2)

{

    latitude1 = ANGLE_TO_RAD(latitude1);

    latitude2 = ANGLE_TO_RAD(latitude2);

    longitude1 = ANGLE_TO_RAD(longitude1);

    longitude2 = ANGLE_TO_RAD(longitude2);

         

    double x = func_sin(longitude2 - longitude1)* func_cos(latitude2);

    double y = func_cos(latitude1)* func_sin(latitude2)- func_sin(latitude1)* func_cos(latitude2)* func_cos(longitude2 - longitude1);

    double angle = RAD_TO_ANGLE(atan2f(x, y));

    return((angle >0)? angle :(angle +360));

}

         

这样计算出来的角度就是GPS与陀螺仪的正方向偏差，然后记录下这个偏差值，后面GPS计算的角度全部减去这个偏差即可与陀螺仪方向角同步。

第二种，以GPS的方向角为主。由于车模静止不动时，方向角不稳定，就让车模先向前行驶，一段时间后，GPS的方向角会趋向稳定。然后让陀螺仪的初始角度等于GPS的方向角，以这个方向角为基准，来调整车模的方向。

         

3.4.4、导航

首先记录路径，沿着需要经过的路径走，每隔3m - 6m记录一个经纬度信息。可以保存到单片机的flash中。

和上一步计算角度相同，计算第一个点与自身当前位置的角度偏差，因为已经同步了方向角，只需计算与车模自身方向角相差来控制车模舵机。能看到车模前轮会指向目标位置，这个时候给一定速度，车模便会向目标位置行驶。

由于不止一个目标点，接近目标点后，需要将目标设置为下一个目标点。用到GPS库中的距离计算。

         

//-------------------------------------------------------------------------------------------------------------------

// 函数简介     计算从第一个点到第二个点的距离

// 参数说明     latitude1       第一个点的纬度

// 参数说明     longitude1      第一个点的经度

// 参数说明     latitude2       第二个点的纬度

// 参数说明     longitude2      第二个点的经度

// 返回参数     double          返回两点距离

// 使用示例     get_two_points_distance(latitude1_1, longitude1, latitude2, longitude2);

// 备注信息

//-------------------------------------------------------------------------------------------------------------------

double get_two_points_distance (double latitude1,double longitude1,double latitude2,double longitude2)

{

    constdouble EARTH_RADIUS =6378137;                                        // 地球半径(单位：m)

    double rad_latitude1;

    double rad_latitude2;

    double rad_longitude1;

    double rad_longitude2;

    double distance, distance1, distance10, distance11, distance12, distance13, distance14, distance2, distance3;

    double a;

    double b;

    rad_latitude1 = ANGLE_TO_RAD(latitude1);                                    // 根据角度计算弧度

    rad_latitude2 = ANGLE_TO_RAD(latitude2);

    rad_longitude1 = ANGLE_TO_RAD(longitude1);

    rad_longitude2 = ANGLE_TO_RAD(longitude2);

         

    a = rad_latitude1 - rad_latitude2;

    b = rad_longitude1 - rad_longitude2;

         

    distance =2* asinf(func_sqrt(func_pow(func_sin(a /2),2)+ func_cos(rad_latitude1)* func_cos(rad_latitude2)* func_pow(func_sin(b /2),2)));   // google maps 里面实现的算法

    distance = distance * EARTH_RADIUS;

         

    return distance;

}

         

这里有一个细节需要注意，在行驶中实时计算自身位置和目标位置的距离，可以设置一个阈值，如2m，当车模当前位置与目标位置的距离小于2m时，将目标点向下一个目标切换，这样可以有效避免更新延迟+小车本身转向半径受限，总是从旁边路过，从而带来的一直到达不了目标点尴尬，其实也相当于一个简单的滤波。
